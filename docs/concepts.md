# Ludoforge 기본 개념 — 일반 프로그래머용 설명

> 이 문서는 Ludoforge가 다루는 문제와 거기 쓰이는 용어를, SMT/형식검증을
> 처음 접하는 일반 프로그래머도 이해할 수 있게 설명한다.
> 프로젝트의 의사결정 SSOT는 [CLAUDE.md](../CLAUDE.md), 구현 계획은 [PLAN.md](../PLAN.md).

---

## 1. 한 줄 요약

여러 기획자가 따로 쓴 게임 룰을 **DSL**로 정형화하고, 이를 **Z3(SMT solver)** 로
번역해 룰들 사이의 **논리적 모순을 (우연이 아니라) 증명으로** 찾아내는 도구다.

---

## 2. 왜 필요한가 — 문제부터

기획자 A: "전사 최대 HP = 레벨 × 100"
기획자 B: "모든 캐릭터 HP는 5000을 넘지 않는다"
기획자 C: "레벨 상한은 100"

세 룰 각각은 합리적이다. 그런데 **함께 두면** 레벨 51 전사는 HP가 5100이어야
하므로 상한 5000과 충돌한다. 즉 "레벨 51 이상 전사"는 존재할 수 없는 상태가 된다.

사람이 눈으로 모든 룰 조합을 검토하는 건 룰이 수백 개면 불가능하다.
**시뮬레이션** 도구는 우연히 레벨 51 전사를 만들어봐야 버그를 본다.
Ludoforge는 그런 모순이 **반드시 존재함을 수학적으로 증명**한다.

---

## 3. 핵심 용어 사전

### DSL (Domain-Specific Language, 도메인 특화 언어)
특정 문제 영역만을 위한 작은 언어. 여기선 게임 룰을 산문(자연어) 대신
기계가 읽을 수 있는 구조로 적는 형식(초기엔 YAML 기반).
**정형화하는 행위 자체가** "전사가 뭐다", "HP 상한이 뭐다" 같은 암묵 가정을
드러내게 만드는 게 핵심이다.

### SSOT (Single Source of Truth, 단일 진실 원천)
"이 정보의 정답은 여기 하나뿐"이라고 약속한 위치. Ludoforge에선 룰의 SSOT가
산문 기획서가 아니라 **DSL 파일(`.rule`)** 이다. git으로 버전 관리한다.

### IR (Intermediate Representation, 중간 표현)
DSL 텍스트를 곧장 Z3로 보내지 않고, 중간에 두는 프로그램 내부 자료구조
(파이썬 데이터클래스). `텍스트 → IR → Z3` 단계를 나누면 검증·테스트·번역이
각각 단순해진다. 컴파일러가 소스코드를 AST로 바꾸는 것과 같은 발상.

### SMT solver / Z3
- **SAT solver**: "이 불리언 논리식을 참으로 만드는 변수 할당이 존재하는가?"를 푸는 도구.
- **SMT (Satisfiability Modulo Theories)**: SAT을 확장해 정수·실수·산술 같은
  "이론(theory)"까지 다루는 것. `x + y <= 10 ∧ x > 3` 같은 식을 풀 수 있다.
- **Z3**: Microsoft Research가 만든 대표적 SMT solver. RuleForge의 판정 엔진.
  **모순 판정은 항상 Z3가 한다 — LLM이 아니다.** (LLM은 거짓 일관성을 환각할 수 있음)

### sat / unsat / unknown — Z3의 세 가지 답
제약식 묶음을 Z3에 넣으면 셋 중 하나를 답한다:
- **sat (satisfiable)**: "모든 제약을 동시에 만족하는 변수값이 존재한다." → 그 예시 값(model)을 준다.
- **unsat (unsatisfiable)**: "어떤 값으로도 전부 만족시킬 수 없다." → 제약들이 서로 **모순**.
- **unknown**: 시간초과나 이론적 한계(아래 NIA 참고)로 **판단 못 함**.
  Ludoforge는 이걸 절대 sat/unsat으로 뭉개지 않고 별도로 보고한다.

### model (모델) / 반례
sat일 때 Z3가 주는 "이렇게 하면 다 만족됨"의 구체적 변수 할당.
"위반 상태가 존재하는가?"를 물어 sat이 나오면, 그 model이 곧 **반례**
(룰이 깨지는 입력 예시)다. 디버깅에 직접 쓰인다.

### assert_and_track / unsat core (이 프로젝트의 존재 이유)
- Z3에 제약을 등록할 때 그냥 넣으면(익명 assert) unsat이 나와도 **어떤 제약이
  범인인지** 모른다.
- `assert_and_track(제약, "rule_id")` 로 **이름표를 붙여** 등록하면, unsat일 때
  Z3가 `unsat_core()` — 즉 **모순을 일으킨 최소 룰 집합**을 ID로 돌려준다.
- 위 예시라면 core = `{warrior_hp_formula, global_hp_cap, level_cap}`.
  "이 세 룰이 서로 싸운다"를 정확히 짚는 것. → **그래서 익명 assert는 금지.**

### Implies — 조건부 룰 표현
`when`/`then` 구조의 룰은 논리 함의 `Implies(조건, 결과)`로 번역된다.
`role == warrior → hp == level * 100`. 조건이 거짓이면(마법사면) 그 룰은
자동으로 "만족"으로 처리돼 영향을 안 준다.

### enum 인코딩 (EnumSort)
`role: [warrior, mage, archer]` 같은 열거형을 Z3의 **EnumSort**(전용 타입)로 만든다.
각 enum이 별도 타입이라 값이 타입으로 구별된다 — 서로 다른 enum이 같은 값 이름(예: 둘 다
`open`)을 써도 안전하고, 무의미한 순서 비교(`role < mage`)는 타입상 막힌다.
(초기엔 정수 0,1,2…로 인코딩했으나 D8에서 EnumSort로 교체. 자세히는 아래 §7.)

### LIA / LRA / NIA — 산술의 종류와 난이도
Z3가 다루는 산술 "이론"의 구분. 여기서 성능과 결정가능성이 갈린다:
- **LIA** (Linear Integer Arithmetic, 선형 정수): `hp == level * 100` 처럼
  변수에 **상수**만 곱하는 식. 빠르고 항상 결정 가능. 1차 핵심 범위.
- **LRA** (Linear Real Arithmetic, 선형 실수): 확률 합 = 1 같은 비율/실수 계산.
  2차(D7)에서 `type: real`로 지원. 나눗셈은 상수 분모만(`1/3`은 정확한 유리수).
- **NIA** (Nonlinear Integer Arithmetic, 비선형): `atk * crit_mult`처럼
  **변수 × 변수**. 느리거나 아예 결정 불가(unknown)가 될 수 있다. → **피해야 함**
  (한쪽을 상수화하거나 구간 분할로 우회, 안 되면 한계를 리포트에 명시).

## 4. 핵심 통찰: "그냥 다 합쳐서 unsat 검사"로는 부족하다

가장 중요한 설계 포인트다. 위 warrior 예시의 룰을 전부 Z3에 넣고 unsat을
물으면 답은 **sat**이다 — 모순이 없다고 나온다. 왜?

> Z3가 `role = mage, level = 100` 또는 `role = warrior, level = 50` 처럼
> **모순을 피해 가는 값**을 하나 찾아버리면 그걸로 "만족 가능"이 되기 때문이다.

진짜 모순은 **조건부**다: "전사가 레벨 100까지 도달 가능해야 한다"는 기획자의
의도가 룰에는 안 적혀 있고 `level.max = 100`에 암묵적으로만 있다.
모순의 본질은 **"기획자가 합법이라 여기는 상태를, 룰들이 불가능하게 만든다"** 이다.

### 그래서 — 도달 가능성(reachability) 검사
"선언된 도메인 전 구간이 룰 하에서 실제 달성 가능한가?"를 묻는다.
구현은 Z3의 **Optimize**(최적화: 제약 하에서 변수의 최대/최소를 구함)로 한다:

1. 각 enum 값 고정(`role == warrior`).
2. 각 수치 변수에 대해 룰 하에서 **실제 달성 가능한 max/min**을 Optimize로 구함.
3. 그게 **선언된 범위보다 좁으면** = 룰이 도메인 일부를 봉쇄 = **모순**.
   → "전사는 레벨 50까지만 가능(선언 max=100)" + 범인 룰 ID 보고.

이 방식은 비용이 `enum 값 수 × 변수 수`로 선형이라 조합 폭발이 없다.

## 5. 검증 파이프라인 한눈에

```
.rule (DSL, YAML, git SSOT)
   │  ① 파서/로더        → IR(중간표현)
   │  ② 스키마·참조 검증  → 미정의 심볼·순환의존 등 형식 오류 (실패 시 여기서 중단)
   │  ③ Z3 번역기        → 제약식, assert_and_track으로 룰마다 이름표
   │  ④ 검사 실행        → 도달성(Optimize) / 모순(unsat + unsat_core) / 반례(model)
   │  ⑤ 리포터          → unsat core·반례를 한국어 리포트로
   ▼  ⑥ CI 통합         → PR마다 자동 실행, 모순이면 fail
```

**두 검사를 분리**하는 게 원칙이다: 형식/참조 무결성(②, 스키마)이 통과해야
논리 모순 검사(④, Z3)로 넘어간다.

---

## 6. 역할 경계 — LLM은 무엇을 하고 안 하나

- **판정(모순 여부)은 무조건 Z3.** 결정론적이어야 하므로 LLM에 안 맡긴다.
- LLM/AI는 **보조만**: (a) 산문 → DSL 번역 보조, (b) unsat core를 사람 말로 설명,
  (c) 신규 룰의 잠재 충돌 후보 제시.
- "이 룰들 모순 같은데?" 라는 LLM 직관으로 결론짓지 않는다. 반드시 Z3에 넣어
  `unsat`을 근거로 삼는다.

## 7. 2차 확장 — 표현할 수 있는 룰과 검사의 폭

1차는 정수 수치공식 + 조건부(`when`) + enum + 도달성(Optimize)이었다.
2차에서 다음을 더했다. 각 항목 끝의 D번호는 설계 결정 기록([decisions.md](decisions.md)).

### 불리언 상태와 상호 배제 (D6)
"은신 중", "공격 중" 같은 on/off 상태를 `type: bool` 변수로 둔다. `not (stealthed and
attacking)` 같은 룰로 "동시에 참일 수 없다"(상호 배제)를 적는다. 검사는 도달성의 연장이다:
**자유 bool의 True/False 각 상태가 룰 하에서 가능한지** 본다. 어떤 상태에 영영 못 들어가면
(예: "항상 공격" 룰 + 상호 배제 → 은신 불가) 모순으로 보고한다. 기획자가 무조건 고정한
bool(예: `then: "stealthed"`)은 의도된 것이라 검사에서 제외한다(거짓양성 회피).

### 비율·확률과 실수 산술 LRA (D7)
확률 합 = 1 같은 룰엔 실수가 자연스럽다. `type: real`로 실수 변수를 둔다(Z3 Real). 1차엔
정수 퍼센트(0~100)로 스케일링해 우회했는데 이제 직접 쓴다. `1/3`은 부동소수 0.333…이 아닌
**정확한 유리수**로 다룬다(상수 분모만 허용 — 변수 분모는 비선형이라 거부). 대표 모순은
over-constraint: 합=1인데 각 확률 하한이 너무 커 어떤 값으로도 합 1을 못 맞추는 경우.

### enum을 EnumSort로 — 타입 안전과 중복 이름 (D8)
1차는 enum을 정수(warrior=0…)로 인코딩했다. 그러면 `role < mage` 같은 무의미한 순서
비교가 우연히 허용되고, 서로 다른 enum이 같은 값 이름을 쓰면 충돌했다. Z3 **EnumSort**는
각 enum을 별도 타입으로 만든다 — 값이 타입으로 구별돼 같은 이름도 안전하고, 잘못된 비교는
타입 오류로 걸린다. `role == warrior`의 `warrior`는 비교 상대(`role`)의 타입으로
해석한다(문맥 기반).

### 실수 변수의 끝점 도달성 (D9)
정수는 Optimize로 "달성 가능한 최대/최소"를 정확히 구해 선언 범위와 비교한다. 실수는
Optimize가 비-도달 상한(`x < 1`이면 1에 무한히 접근만)을 무한소(epsilon)로 돌려줘 다루기
까다롭다. 그래서 실수는 더 가벼운 방식을 쓴다: **선언 끝점이 도달 가능한지**(`var == 끝점`이
sat인지)만 본다. "크리율을 [0,1]로 선언했지만 룰이 0.5로 막음 → 1.0 도달 불가" 같은 봉쇄를
잡는다.

### 명시적 도달성 단언 `expect:` (D10)
지금까지 도달성은 **선언 도메인에서 자동 추론**했다. 그런데 "공격과 방어를 **동시에** 최대로
찍을 수 있어야 한다" 같은 **변수 조합의 도달성**은 변수별 경계로는 안 보인다(각각은 단독으로
최대에 도달하니까). `expects:` 섹션에 기획자가 "이 상태는 가능해야 한다"(`that`)를 직접
적으면, `룰 ∧ that`가 sat인지 확인한다 — unsat이면 룰이 그 조합을 막은 것이라 모순으로
보고하고 범인 룰을 짚는다. 자동 추론의 **역방향**이다.

## 8. 3차 확장 — 다중 백엔드: 동역학과 확률 (D11~D16)

여기까지(1·2차)는 변수들의 **단일 스냅샷**에서 논리 모순을 봤다. 3차는 차원을 둘 더했다:
**시간(전이)** 과 **확률**. 동기는 실제 질문이었다 — "이 도구로 보드게임 *던전!*(WotC)을
검증할 수 있나?" 던전!은 턴마다 이동하고 주사위로 싸우고 골드를 모아 귀환하는 게임이라,
정적 스냅샷으론 표현조차 안 됐다. 그래서 전이 시스템을 도입하고, 검증 엔진을 둘로 늘렸다.

### 8.1 왜 또 확장? — 정적 스냅샷의 한계

1·2차의 룰은 "한 순간 상태가 합법인가"만 말한다. 하지만 게임은 **상태가 시간에 따라
변한다**: 골드가 턴마다 쌓이고, 방을 옮기고, 전투 결과가 갈린다. "10턴 안에 승리에
도달 가능한가?", "어떤 진행에서도 골드가 음수가 안 되는가?", "승리 확률은?" 같은 질문은
스냅샷 한 장으로 못 푼다. **시간**과 **확률**이 필요하다 — 그런데 이 둘은 서로 다른 수학
이라, 한 엔진(Z3)으론 다 못 한다. 그래서 **모델은 하나, 백엔드는 둘**로 간다(D11).

### 8.2 전이 시스템 — 상태가 변하는 모델

DSL에 세 가지를 더했다(D12). 정적 `domain`/`rules`/`expects`는 그대로 유지된다(하위 호환).

```yaml
init: "gold == 0 and room == center"     # 초기 상태 술어
transitions:                              # 상태 → 다음 상태 규칙
  - id: fight
    when: "room == l1"                    # 가드(이 조건일 때만 발생)
    outcomes:                             # 확률 분기
      - { weight: 0.7, then: "next.gold == gold + 5" }   # 70% 승리
      - { weight: 0.3, then: "next.gold == gold" }       # 30% 무소득
properties:                               # 검증하고 싶은 질문
  - { id: winnable, kind: reachable, that: "gold >= win_gold and room == center" }
  - { id: gold_ok,  kind: invariant, that: "gold >= 0" }
```

- **`next.<변수>`**: *다음* 상태의 값. `next.gold == gold + 5`는 "다음 턴 골드 = 지금 골드
  +5". `next.`는 전이 안에서만 쓸 수 있다(다른 곳에서 쓰면 스키마 오류).
- **`outcomes` + `weight`**: 한 전이가 확률적으로 갈리는 분기. 가중치는 확률 백엔드용이다.
- **프레임 규칙(중요, D15)**: 전이가 `next.X`로 **건드리지 않은** 변수는 다음 상태에서
  **값이 유지된다**. `fight`는 gold만 바꾸므로 room·role은 그대로다. "안 건드리면 그대로"가
  기본 — 안 그러면 캐릭터가 룸을 순간이동하는 식의 헛것이 나온다. (이게 PRISM 갱신 의미와
  같아서 두 백엔드가 모델을 공유할 수 있다.)
- **정적 `rules`는 모든 상태의 불변식(D15)**: `role==wizard → win_gold==20` 같은 룰은 매
  순간 성립해야 하므로, 모든 스텝에 적용된다.

### 8.3 새 용어 사전

#### 전이 시스템 (transition system)
"상태"와 "상태에서 상태로 가는 규칙"의 모음. 게임·프로토콜·회로 등 시간에 따라 변하는
시스템의 표준 수학 모델. `초기 상태 + 전이 관계`로 정의된다.

#### 비결정 (nondeterminism)
한 상태에서 갈 수 있는 다음 상태가 **여럿**인 것. 던전!에선 (a) 여러 전이가 동시에 가능
하거나, (b) 자유 변수(어떤 직업으로 시작?)가 미정이거나, (c) 주사위 결과가 갈린다. 논리
백엔드는 확률을 **비결정으로** 본다(아래 weight-erasure).

#### 모델 검사 (model checking)
전이 시스템이 어떤 **속성**(도달성·안전성 등)을 만족하는지 **자동·망라적으로** 확인하는
기법. 테스트가 *일부* 경우를 보는 것과 달리, 모델 검사는 *모든* 가능한 진행을 따진다.

#### BMC (Bounded Model Checking, 유계 모델 검사)
모델 검사를 **k 스텝까지만** 펼쳐 SMT(Z3)로 푸는 방식. 상태를 시간별로 복제해
(`gold@0, gold@1, …, gold@k`) "초기 ∧ 전이 ∧ 속성"을 한 식으로 만들어 sat/unsat을 묻는다.
k까지는 **증명**, k 너머는 모른다 — 이 한계(k-bound)를 정직하게 보고한다.

#### PCTL / Pmax / Pmin
확률 시제 논리. `P>=0.95 [ F win ]` = "결국(F) 승리할 확률이 0.95 이상". 비결정이 있는
모델(MDP)에선 스케줄러(전략)에 따라 확률이 달라져 **Pmax**(최선 전략의 확률)·**Pmin**
(최악)으로 나눠 묻는다. `Pmax=? [...]`는 그 최대 확률 **값**을 계산하라는 질의.

#### MDP / DTMC
확률 모델의 종류. **DTMC**(이산시간 마르코프 체인) = 확률만 있고 선택은 없음. **MDP**
(마르코프 결정 과정) = 확률 + 비결정 선택. 던전!은 직업 선택·전이 다중 활성 때문에 MDP다.

#### 상태 폭발 (state explosion)
전이 시스템의 상태 수가 변수 범위의 **곱**으로 늘어 폭발하는 현상. `gold[0..30000] ×
win_gold[0..30000] × room × role` ≈ 70억. 확률 백엔드(PRISM)는 상태공간을 통째로 빌드
하므로 이게 천장이다. (BMC는 k 스텝만 펼쳐 영향이 적다 — 백엔드별 약점이 다르다.)

### 8.4 RuleForge의 BMC 백엔드 — 논리로 동역학 검증 (D15)

`ludoforge bmc <파일> --k N`. 전이를 k 스텝 펼쳐 Z3로 세 가지를 본다:

- **reachable**: 목표 상태에 **도달 가능한가?** sat이면 **도달 경로**(어느 전이를 밟아
  거기 갔는지)를 반례로 제시. (예: 던전! 승리는 깊이 4에 도달.)
- **invariant**: 불변식이 **항상 유지되는가?** 어느 깊이서 깨지면 **깨짐 시퀀스**를 제시,
  k까지 안 깨지면 "k까지 유지"(유계 보장).
- **no_deadlock**: 어떤 전이도 못 쓰는 **막힌 상태**에 도달하는가?

**주사위 = 비결정(weight-erasure)**: BMC는 확률 가중치를 **버리고** "분기 중 아무거나
가능"으로 본다. 그래서 reachable은 "*어떤* 주사위열이 승리를 만든다"를, invariant 위반은
"*어떤* 주사위열이 규칙을 깬다"를 뜻한다. 이건 시뮬레이션이 못 하는 걸 준다 — 나쁜
경우를 *우연히 만나길* 기다리지 않고 **존재를 증명**하거나, 모든 주사위열에서 안전함을
보장한다.

### 8.5 ProbForge 백엔드 — 확률로 동역학 검증 (D16)

`ludoforge prob <파일>`. 같은 전이 시스템을 **PRISM**(확률 모델 검사기)용 모델로 번역해
**정량** 질문에 답한다 — BMC가 못 보는 "승리 *확률*", "기대 게임 길이".

- IR을 PRISM guarded-command 모델로 번역: enum은 정수 const로, 정적 rules+init은 초기
  상태 술어로, 전이는 확률 명령으로(가중치 정규화), 속성은 PCTL로(reachable→`Pmax=? [F]`,
  invariant→`Pmin=? [G]`).
- **유한 상태가 전제**: PRISM은 상태공간을 빌드하므로 경계 없는 int·실수를 거부한다
  (`check_finite_state` 게이트, D13). 무한을 허용하는 Z3와 정반대 제약.
- 던전! 검증 결과: 승리 확률 `Pmax [F win] = 1.0` (성공률 0.7로 무제한 재도전하면 결국
  승리). 단, 큰 수치는 상태 폭발을 부르므로 예제는 `gold[0..20]`로 축약했다.
- **확률 vs 비결정**: PRISM은 가중치(0.7/0.3)를 **살려** 실제 확률을 계산한다. 같은
  전이를 BMC는 비결정으로(버리고), ProbForge는 확률로(살려) 본다 — 정성 모델은 정량
  모델에서 확률을 *잊은* 건전한 추상이라, 한 DSL이 둘을 모순 없이 담는다.

### 8.6 다중 백엔드 아키텍처 — 모델은 하나, 질문은 여럿

```
            forge_core/   ← 공유 DSL 프론트엔드(SSOT): 로더·스키마·IR
             (전이 시스템: 변수·init·transitions·properties)
                  │
        ┌─────────┴─────────┐
        ▼                   ▼
    ruleforge/          probforge/
   IR → Z3 / BMC        IR → PRISM
   논리 증명             확률 증명
   (도달성·불변식·       (승리 확률·기대값,
    데드락, 반례 경로)    PCTL)
```

핵심 설계 원칙(D11):
- **파서·IR은 공유 라이브러리(`forge_core`).** 두 백엔드가 *같은* IR을 소비한다. 각자
  파서를 따로 두면 SSOT가 둘로 갈라져 어긋난다 — 그래서 하나로 묶는 게 생명선이다.
- **모델은 공유, 질의(속성) dialect는 백엔드별.** 전이 모델은 둘이 같지만, "무엇을
  묻는가"의 언어는 다르다(Z3의 도달성/unsat-core vs PRISM의 PCTL). 질의 언어를 억지로
  통일하지 않는다.
- **상보적 약점.** Z3는 무한·정확 산술에 강하나 비선형에 약하고, PRISM은 유한·비선형엔
  강하나 상태 폭발에 약하다. 한 엔진으로 다 하려 했으면 충돌했을 약점이, 나누니 각자
  강점만 쓴다.

### 8.7 던전!으로 본 전체 그림

하나의 `.rule` 파일(`examples/dungeon.rule`)에서:
- `ludoforge check` → 정적 모순 없음(직업별 승리 목표가 충돌 안 함).
- `ludoforge bmc … --k 10` → **승리 도달 가능**(깊이 4 경로 제시), **골드 비음수 유지**.
- `ludoforge prob …` → **승리 확률 1.0**(PRISM 계산).

대화에서 시작된 질문 "던전!을 검증할 수 있나?"의 답이다 — **논리는 Z3/BMC로 증명하고,
확률은 PRISM으로 계산하되, 둘 다 하나의 DSL에서.** (밸런스 *튜닝*·재미 평가는 여전히
비목표 — 그건 Machinations 같은 시뮬레이션 영역이다.)
